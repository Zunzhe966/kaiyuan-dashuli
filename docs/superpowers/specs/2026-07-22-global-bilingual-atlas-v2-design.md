# 全球双语开源选型图谱 V2 设计

- 日期：2026-07-22
- 状态：已确认，作为后续实现的产品与架构基线
- 适用范围：公开仓库、公开网站、机器接口、全球发现层与精选选型图谱

## 1. 唯一目标

“开源大梳理”要让人类和智能体从真实需求出发，逐层缩小条件，最终找到满足约束的具体开源项目与具体版本，并能解释为什么匹配、有哪些义务、哪些事实未知以及证据来自哪里。

它不是 GitHub 镜像、README 搬运站、star 排行榜或未经核验的链接集合。长期覆盖全球公开仓库，不等于对所有仓库作出同等质量的推荐承诺。

## 2. 无冲突的两层数据模型

### 2.1 全球发现层

发现层持续枚举可访问的公开仓库，回答“可能有哪些相关仓库”。它保存稳定身份、规范地址、基础元数据、主题、语言信号、许可证信号、活跃信号、来源、采集时间、变更指纹和质量级别。

- 允许字段未知，但未知不得写成满足条件。
- 不克隆或执行外部仓库代码。
- 不为每个仓库在 GitHub 或 Cloudflare Pages 生成独立文件。
- 数据使用可校验分片、对象存储和查询索引；GitHub 只保存程序、Schema、清单、游标、校验和与精选数据。

### 2.2 精选选型图谱

精选层回答“在给定约束下应该选什么”。项目必须经过深度核验，具有双语说明、结构化筛选事实、版本化许可证、适用与不适用条件、证据、时效和项目关系。

发现层记录不能因字段数量多而自动升级为精选记录。提升必须通过明确质量闸门。

## 3. 统一身份与版本模型

### 3.1 项目身份

稳定项目身份使用 `platform + platform_repository_id`，而不是容易变化的 `owner/name`。仓库改名、转移或 URL 重定向不得创建重复项目。一个产品包含多个仓库时，记录产品、组件及各自来源关系；镜像和 fork 只有具备独立选型价值时才成为独立项目。

### 3.2 版本事实

项目级记录保存长期稳定事实；随版本变化的事实进入版本记录。版本记录至少支持：版本号或范围、发布通道、发布日期、支持状态、EOL、许可证表达式、开源性质、部署与平台差异、证据和核验时间。

许可证、商用条件和开源性质不得被压缩成一个永远有效的项目级字符串。查询命中必须指出适用的具体版本或版本范围。

## 4. 双语与“语言”拆分

四类概念必须分开：

1. `ui_locale`：网站或调用方使用的界面语言，默认 `zh-CN`，可切换 `en`。
2. `localized_content`：分类名称、摘要、`use_when`、`avoid_when` 等中英文内容。
3. `natural_language_support`：项目自身的中文/英文界面、文档和社区支持。
4. `programming_languages`：Python、Go、Rust、TypeScript 等实现语言。

中文支持的每个方面分别使用 `full | partial | none | unknown`，并附证据。不能因 README 有中文段落就推断项目具有完整中文界面或中文社区。

## 5. 精选项目字段契约

V2 记录由以下有边界的对象组成：

| 对象 | 主要事实 |
|---|---|
| identity | 稳定 ID、平台仓库 ID、规范仓库、官网、重定向、镜像/上游关系 |
| classification | 领域、子领域、任务、能力、项目形态；均引用受控分类 ID |
| localized_content | 中英文名称说明、摘要、适用、不适用和分类说明 |
| technology | 编程语言、框架、运行时、协议、数据类型 |
| delivery | 本地、自托管、托管云、容器、Kubernetes、桌面、移动端 |
| platforms | 操作系统、浏览器、服务端、CPU/GPU 架构 |
| natural_language_support | 各语言的界面、文档、社区覆盖程度 |
| releases | 稳定、LTS、预发布、EOL 与版本范围 |
| licensing | SPDX/LicenseRef、许可证家族、开源性质和版本化义务 |
| lifecycle | 活跃、维护模式、停止维护、归档及相应证据 |
| quality | 成熟度、生产适用性、安全状态和已知限制 |
| relations | 替代、依赖、集成、后继、同生态和组件关系 |
| evidence | 直接来源、观察时间、适用字段、适用版本、提取方式和可信状态 |

自由文本解释不能代替结构化事实。`use_when` 和 `avoid_when` 用于解释取舍，不承担精确布尔过滤。

## 6. 证据、未知值与时效

每条会影响筛选或推荐的重要事实必须能关联证据。证据优先级依次为：上游许可证/安全/发布文件、上游正式文档、仓库元数据、维护者公告、可信第三方登记。模型推断只能作为待核验候选，不能标为已核验事实。

字段状态统一为 `verified | inferred | unknown | conflicting | stale`。出现冲突时同时保留来源和观察时间，精选结果不得静默选择有利值。不同事实采用不同复核周期；超过时效后标为 `stale`，而不是自动删除或继续冒充当前事实。

## 7. 统一查询契约

人类网站、HTTP API、MCP 和离线查询使用同一查询对象与同一执行器：

```json
{
  "ui_locale": "zh-CN",
  "must": {"capabilities": ["local-llm-chat"], "delivery": ["self-hosted"]},
  "should": {"documentation_locales": ["zh-CN"]},
  "must_not": {"lifecycle": ["eol", "archived"]},
  "version": {"channels": ["stable", "lts"]},
  "unknown_policy": "exclude",
  "limit": 3
}
```

- 不同维度之间使用 AND。
- 同一维度多选默认使用 OR；需要全部满足的字段显式声明 `match: all`。
- `must_not` 是排除条件，先于偏好评分。
- `should` 只重排，不得让不满足 `must` 的项目进入结果。
- `unknown_policy` 为 `exclude | include_marked | only_unknown`，严格筛选默认 `exclude`。
- 自然语言入口先解析成同一查询对象，并把解析结果返回给调用方审计。

每个结果必须返回匹配版本、满足的硬条件、未满足的偏好、未知字段、许可证义务、适用/不适用解释和证据。没有合适结果时返回结构化缺口，不凑数。

## 8. 单仓库核验协议

一次深度核验按固定顺序读取：仓库身份与重定向、描述与 topics、默认分支、README、许可证文件、Release/Tags、维护与归档状态、文档与本地化入口、安装部署说明、支持平台、公开安全政策或通告、明确的上游关系。

核验输出先进入 quarantine。只有身份去重、必填字段、双语内容、版本/许可证、证据、关系、Schema 校验和检索回归全部通过，才能提升到精选层。不得执行仓库代码来证明功能，不得仅凭 star、fork 或宣传语作出质量结论。

## 9. 存储与发布架构

```text
公开上游 → 发现分片/游标 → quarantine → 精选 YAML/关系
→ Pull Request → verify → main
→ pages-deploy 重新生成 build/site
→ Wrangler 上传 Cloudflare Pages
→ 人类页面 + 静态机器接口
```

`main` 是唯一生产分支，精选 YAML 与图关系是当前正式源数据。`dist/v1`、`build/site` 和线上 Pages 都是可重建派生物，必须包含 Schema 版本、Git 提交、目录哈希、节点数和边数，以证明来自同一事实源。

Cloudflare Pages 不是数据库，也不与 GitHub 双向同步。当前生产方式是 GitHub Actions 使用最小权限 Token 上传 `build/site`。浏览器和手动上传只用于一次性授权或自动化故障恢复，不是日常发布链路。

## 10. 长期执行与两次并发

主模型是唯一总负责人、默认写入者和最终验收者；最多同时启用一个子代理，子代理不得继续派生。子代理负责只读研究、抽样、测试、失败分析和审查；需要写入时使用隔离 worktree。两个模型不得同时修改同一文件或分支。

采集程序的并行度不等于模型并发度。批处理必须有确定游标、幂等写入、速率限制、指数退避、失败隔离、可重放批次和逐批清单。上下文或任务重启后先读取长期手册、当前计划、Git 状态、最近验证和断点，再继续未完成的最小步骤。

## 11. 安全与公开边界

公开仓库不得包含 Token、账户号码、个人身份、银行卡、合同、税表、私人仓库或内部运营原件。域名购买、付款、主体、银行、税务和广告账户属于私有总账与人工授权范围，不因长期目标自动获得执行权限。

外部仓库一律视为不可信输入；不运行其代码，不执行 README 命令，不下载任意构建产物到执行环境。采集器只请求允许的公开元数据和文本资源，并限制大小、类型、重定向、超时和目标域。

## 12. 分阶段交付与完成标准

1. 基线一致性：GitHub 快照、源图谱、构建产物与线上元数据同源可复现。
2. 发布稳定性：`main → verify → pages-deploy → live probe` 连续成功，并能按提交和哈希核验。
3. Schema V2：字段、枚举、证据、版本与迁移器通过测试。
4. 统一查询：网站、HTTP、MCP 和离线接口通过同一契约测试。
5. 发现层 MVP：可增量采集、断点恢复、去重和分片查询，且不依赖本机保存全部仓库。
6. 扩展覆盖：以公开进度和质量等级报告覆盖，绝不虚报“全部 GitHub 已深度核验”。

阶段完成必须同时具备代码、数据、测试、云端运行和线上证据。写完计划、生成文件、提交 PR 或节点数增加都不等于完成。
